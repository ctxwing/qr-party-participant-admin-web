# SOS 요청이 관리자가 아닌 모든 사용자에게 전달되는 문제

## 문제 발생 시각
- 2026-04-26 15:00 (KST)

## 문제 증상
- 사용자가 SOS 요청 시 **모든 사용자 대시보드에 팝업**으로 표시됨
- 정상 동작: SOS는 **관리자에게만** 전달되어야 함
- 노래 신청도 관리자에게 전달되지 않고 `interactions` 테이블로 저장됨 (관리자 모니터링 대상 아님)

## 원인 분석

| 항목 | 내용 |
|------|------|
| 근본 원인 | SOS 핸들러가 `announcements` 테이블에 INSERT → 모든 사용자의 Realtime 리스너가 `announcements` INSERT를 수신 |
| 구조적 문제 | 테이블 용도 구분 없이 사용자→관리자, 관리자→사용자 요청이 모두 같은 테이블 사용 |

### 테이블 용도 정의 (수정 후)

| 테이블 | 방향 | 용도 | Realtime 구독자 |
|--------|------|------|-----------------|
| `alerts` | 사용자 → 관리자 | SOS, 노래신청 | 관리자 페이지만 |
| `announcements` | 관리자 → 사용자 | 공지사항 | 모든 사용자 대시보드 |

## 해결 체크리스트

- [x] 1. SOS: `announcements` INSERT → `alerts` INSERT로 변경
  ```ts
  // 수정 전: 모든 사용자에게 팝업
  await supabase.from('announcements').insert({
    content: `🚨 SOS (${nickname}): ${msg}`, type: 'emergency'
  });

  // 수정 후: 관리자만 모니터링
  await supabase.from('alerts').insert({
    participant_id: participant.id,
    type: 'SOS',
    message: `🚨 SOS (${nickname}): ${msg}`,
    resolved: false
  });
  ```
- [x] 2. 노래신청: `interactions` INSERT → `alerts` INSERT로 변경
  ```ts
  await supabase.from('alerts').insert({
    participant_id: participant.id,
    type: 'MUSIC',
    message: `🎵 노래신청 (${nickname}): ${song}`,
    resolved: false
  });
  ```
- [x] 3. 관리자 admin 페이지에서 SOS/MUSIC alerts 정상 수신 확인

## 참고 사항
- 사용자→관리자 요청은 반드시 `alerts` 테이블 사용 (관리자만 구독)
- 관리자→사용자 공지는 `announcements` 테이블 사용 (모든 사용자 구독)
- 향후 새로운 실시간 기능 추가 시 **메시지 방향(누가→누구에게)** 에 따라 테이블 선택 필요
