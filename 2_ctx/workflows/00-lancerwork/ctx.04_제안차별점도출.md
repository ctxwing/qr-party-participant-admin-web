### 2단계: 현재 구현 및 계획 현황 파악

> **[OO]을 고민하여 → [OOO]할 목적으로 → [OOO]을 반영하여 → [OOO] 효과를 도출**

## 차별점 도출 관점

### 관점 1. 기능적 차별성
경쟁사가 고객 요구사항을 그대로 옮겨 제안할 것에 대비하여, **고객이 요구하지 않았지만 우리가 선제적으로 추가·강화한 기능**을 중심으로 서술한다.
- 서비스 품질 향상을 위해 운영 관점에서 추가한 기능
- 최종 사용자의 편의를 위해 UX 차원에서 보강한 기능
- 고객이 미처 고려하지 못한 엣지케이스에 대한 대응

### 관점 2. 기술적 설계·구현 차별성
안정성, 고성능, 확장성, 보안 등을 위해 **우리가 고민하여 아키텍처·설계에 반영한 기술적 결정**을 서술한다.
- 예) 모바일 반응형 기본 지원, 대용량 트래픽 대비 Queue/Redis 적용, Pre-signed URL 보안 설계 등

### 관점 3. 도메인 경험 기반 선제적 고려
해당 산업·분야의 **다수 프로젝트 수행 경험**을 바탕으로, 경쟁사가 간과하기 쉬운 사항을 우리가 선제적으로 반영한 점을 서술한다.
- 예) "OOO 분야의 OO 서비스 다수 구축 경험을 보유하여, OO 이슈를 사전에 고려함"

## 출력 양식 (A4 반 페이지 이내)

```markdown
# 제안 차별점 (Key Differentiators)

## 1. [차별점 제목]
*   **차별성**: [우리만의 차별적 접근 방식]
*   **효용**: [고객 관점에서의 구체적 기대 효과]

## 2. [차별점 제목]
*   **차별성**: ...
*   **효용**: ...

(3~5개 이내로 핵심만 간결하게)
```

## 주의사항
- 출력 결과는 **A4 반 페이지(2/4)를 초과하지 않는다**.
- 고객의 눈높이에서 이해할 수 있는 언어로 효용을 설명한다. 단, 핵심 기술 용어는 적정 수준으로 사용하되 필요 시 괄호로 쉬운 설명을 병기한다.
- 뻔한 내용(요구사항 그대로 반복)은 차별점이 아니므로 제외한다.
- **과장 표현 절대 금지**: "국내 유일", "완벽", "최초", "보장합니다", "약속합니다" 등 검증 불가능한 수식어를 사용하지 않는다. 사실 기반으로 담백하게 서술한다.
- **적정 수준의 기술 용어 사용**: 임의 조어(예: "1급 시민", "First-Class")는 사용하지 않는다. 업계에서 통용되는 기술 용어(예: API, PDF, 서버, 워터마크 등)는 그대로 사용하되, 과도한 영어 나열은 지양한다. 비전문 독자를 위해 필요 시 괄호로 보충 설명을 덧붙인다.
- **문체**: 모든 서술은 "~함", "~음", "~임" 체로 통일한다. (예: "적용했습니다" → "적용함", "가능합니다" → "가능함")
- **2줄 이내 간결 서술**: 차별성과 효용 각각 최대 2줄을 넘지 않는다. 핵심만 짧게 쓰되, 사용자·관리자 입장에서 체감할 수 있는 효과를 강조한다.
- **이모지·이모티콘 사용 금지**: 문서 내 이모지(🎯, ✅, ⚠️ 등)를 사용하지 않는다. 기호가 필요한 경우 ASCII 특수문자(*, -, >, # 등)와 ASCII 원문자 (①,②, ③, 등). 도형 문자(□, ■, ▣, ◇, ◆, △, ▲, ○, ● 등)는 사용 가능하다.
