## User Scenarios & Testing *(mandatory)*

### User Story 1 - 파티 생명주기 운영 및 실시간 공지 (Priority: P1)

관리자는 파티를 생성하고, 예약자를 연동한 뒤, 파티 시작부터 종료까지의 전 과정을 실시간으로 관리하며 공지사항을 전송할 수 있어야 합니다.

**Why this priority**: 파티 운영의 핵심 기능으로, 실제 이벤트 진행에 필수적인 요소입니다.

**Independent Test**: 파티 생성 후 '시작' 상태로 변경하고, 공지를 발송했을 때 참여자 화면에 즉시 노출되는지 확인하여 테스트 가능합니다.

**Acceptance Scenarios**:

1. **Given** 관리자 대시보드에서, **When** [+ 신규 파티 생성]을 클릭하여 정보를 입력하고 저장하면, **Then** 파티 목록에 '대기' 상태로 표시된다.
2. **Given** 진행 중인 파티 상세 화면에서, **When** 공지 내용을 입력하고 [발송]을 누르면, **Then** 모든 참여자의 모바일 화면에 즉시 공지 팝업이 노출된다.

---

### User Story 2 - 파티 이력 및 데이터 관리 (Priority: P2)

관리자는 과거에 진행했던 파티 이력을 검색하고 상세 참여자 명단 및 활동 내역을 조회할 수 있어야 합니다.

**Why this priority**: 운영 효율성을 높이고 과거 데이터를 분석하여 다음 파티를 준비하는 데 도움을 줍니다.

**Independent Test**: 검색 필터(날짜, 파티명)를 적용했을 때 정확한 결과가 페이징되어 노출되는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 파티 관리 목록에서, **When** 특정 날짜 범위와 파티명을 입력하고 검색하면, **Then** 조건에 맞는 파티 리스트만 노출된다.
2. **Given** 목록에서 [상세보기]를 클릭하면, **When** 해당 파티의 시작/종료 시간 및 참여 인원 상세 정보가 표시된다, **Then** 삭제 버튼 클릭 시 확인 절차 후 데이터가 삭제된다.

---

### User Story 3 - QR 코드 브랜딩 및 설정 (Priority: P3)

관리자는 배포 환경에 맞는 도메인 주소를 설정하고, 로고가 포함된 QR 코드를 생성하여 오프라인 비치를 위해 다운로드할 수 있어야 합니다.

**Why this priority**: 참여자의 접속 경로를 확보하고 브랜드 인지도를 높이는 데 필요합니다.

**Independent Test**: 설정한 URL이 포함된 QR 코드를 스캔했을 때 실제 페이지로 정상 접속되는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** QR 관리 설정 페이지에서, **When** 앵커 URL을 수정하고 저장하면, **Then** 생성된 QR 코드의 스캔 결과가 수정된 URL로 반영된다.
2. **Given** QR 미리보기 화면에서, **When** [PNG 다운로드]를 클릭하면, **Then** 로고와 사이트명이 포함된 QR 코드 이미지가 저장된다.

---

### Edge Cases

- 파티 종료 시간이 시작 시간보다 앞선 경우 시스템은 에러를 발생시켜야 함.
- 실시간 공지 전송 시 네트워크 장애가 발생할 경우 재전송 또는 실패 알림을 표시해야 함.
- 예약 연동 시 중복된 전화번호나 데이터가 있을 경우 처리 정책(덮어쓰기 또는 제외)을 관리자가 선택할 수 있어야 함.
- 동일 앵커 URL로 여러 파티가 생성된 경우, 특정 시점에 '진행 중'인 파티는 오직 하나만 존재해야 하며 중복 활성화는 불가함.
- 동일 데이터를 동시에 수정할 경우 '마지막에 저장한 관리자'의 내용이 최종 반영됨(Last Write Wins).
- 관리자 세션 만료 시 모든 API 요청은 차단되어야 하며 자동으로 로그인 페이지로 리다이렉트되어야 함.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-000**: 시스템은 슈퍼 관리자를 위한 아이디/패스워드 기반의 로그인 기능을 제공해야 한다.
- **FR-001**: 시스템은 파티의 생성, 수정, 삭제 기능을 제공해야 한다.
- **FR-002**: 파티 목록은 페이징(기본 10건) 및 검색 필터(명칭, 날짜)를 지원해야 한다.
- **FR-003**: 파티 상단에 단계별 진행 플로우(Flow Step) 가이드와 상세 매뉴얼 팝업을 제공해야 한다.
- **FR-004**: 관리자는 진행 중인 파티의 참여자에게 실시간(1초 이내)으로 공지 메시지를 전송할 수 있어야 한다.
- **FR-004-1**: 참여자는 실시간 공지 팝업 외에도, 이전에 수신한 공지들을 확인할 수 있는 '공지 이력' 목록을 조회할 수 있어야 한다.
- **FR-005**: 시스템 설정 내 QR 관리 메뉴에서 접속 URL을 자동 로딩하고 편집할 수 있어야 한다.
- **FR-006**: QR 코드는 중앙에 로고를 포함하며, 하단에 사이트명(Bold, 12pt)과 URL(10pt, Gray-600)을 명시해야 한다.
- **FR-007**: 생성된 QR 코드는 이미지 파일(PNG/SVG)로 다운로드 가능해야 한다.
- **FR-008**: [설계만 반영] WordPress 연동은 추후 별도 단계에서 검토한다.
- **FR-009**: 파티 종료 후 해당 QR/URL로 접속 시 "종료된 파티입니다" 안내와 홈 이동 버튼이 포함된 페이지를 노출해야 한다.

### Key Entities *(include if feature involves data)*

- **Party**: 파티 기본 정보(ID, 명칭, 시작/종료 시간, 최대 인원, 상태).
- **Announcement**: 실시간 공지 내역(ID, 파티 ID, 내용, 작성 시간).
- **Anchor URL**: 파티와 연결된 고유 접속 주소 (기간별 재사용 가능).
- **Admin Role**: 시스템 설정 및 파티 생명주기를 관리하는 최상위 권한.

### Constraints & Constitution

- **C-001**: 모든 문서, 코드 주석, 커밋 메시지는 **한글**로 작성하며, 작성자는 **ctxwing@gmail.com**으로 고정한다. (원칙 V, VII)
- **C-002**: 모든 시스템 시간 및 데이터는 **KST (UTC+9)** 및 Docker TZ 설정을 따른다. (원칙 II)
- **C-003**: 오직 **`main`** 브랜치만을 사용하여 개발을 진행하고 완료 내역을 보고한다. (원칙 VII, 거버넌스)
- **C-004**: 핵심 기술 스택(Bun, uv, Next.js 16.2.1+, Drizzle ORM, FastAPI)을 준수한다. (원칙 I, IV)
- **C-005**: UI/UX 규격(ag-grid, Tailwind, shadcn/ui, React Bits, Toast)을 준수한다. (원칙 III)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자가 신규 파티 생성부터 QR 코드 다운로드까지의 과정을 3분 이내에 완료할 수 있음.
- **SC-002**: 실시간 공지 발송 후 100%의 참여자 기기에 1.5초 이내에 메시지가 전달됨.
- **SC-003**: 모바일 기기 3종(iOS, Android 표준) 이상에서 생성된 QR 코드 스캔 시 정상 접속 확인.
- **SC-004**: 1,000건 이상의 파티 이력 데이터 적재 시에도 목록 조회 속도가 0.5초 이내 유지됨.
