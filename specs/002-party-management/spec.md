## User Scenarios & Testing *(mandatory)*

### User Story 1 - 파티 생명주기 운영 및 실시간 공지 (Priority: P1)

관리자는 파티를 생성하고, 예약자를 연동한 뒤, 파티 시작부터 종료까지의 전 과정을 실시간으로 관리하며 공지사항을 전송할 수 있어야 합니다.

**Why this priority**: 파티 운영의 핵심 기능으로, 실제 이벤트 진행에 필수적인 요소입니다.

**Independent Test**: 파티 생성 후 '시작' 상태로 변경하고, 공지를 발송했을 때 참여자 화면에 즉시 노출되는지 확인하여 테스트 가능합니다.

**Acceptance Scenarios**:

1. **Given** 관리자 대시보드에서, **When** [+ 신규 파티 생성]을 클릭하여 정보를 입력하고 저장하면, **Then** 파티 목록에 '대기' 상태로 표시된다.
2. **Given** 진행 중인 파티 상세 화면에서, **When** 공지 내용을 입력하고 [발송]을 누르면, **Then** 모든 참여자의 모바일 화면에 즉시 공지 팝업이 노출된다.

---

### User Story 4 - 참여자 상세 상호작용 관리 (Priority: P1)

관리자는 특정 참여자의 파티 신청 상태를 승인/반려하고, 이벤트 진행 상황에 따라 개별적으로 큐피트나 호감도 발사 횟수를 보너스로 지급하거나 조정할 수 있어야 합니다.

**Why this priority**: 현장 운영의 유연성을 확보하고 참여자의 활동을 실시간으로 독려하기 위해 필수적입니다.

**Independent Test**: 특정 참여자의 큐피트 횟수를 수정했을 때, 참여자의 모바일 화면에 즉시 반영되는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 파티 상세 관리 화면에서, **When** 특정 참여자의 2차 신청 상태를 [승인] 또는 [반려]로 변경하면, **Then** 해당 참여자의 화면에 상태 배지가 즉시 반영된다.
2. **Given** 참여자 목록에서, **When** 특정 인원의 큐피트/호감도 잔여 횟수를 수정하면, **Then** DB와 참여자 대시보드에 즉시 반영된다.

---

### User Story 5 - 통합 모니터링 및 실시간 로그 추적 (Priority: P2)

관리자는 파티 중 발생하는 모든 상호작용(쪽지, 호감도 등)과 닉네임 변경 이력을 실시간 피드로 모니터링하여 이상 징후를 감지하고 운영에 참고할 수 있어야 합니다.

**Why this priority**: 안전한 파티 운영과 사용자 간 분쟁 예방, 활동성 체크를 위해 필요합니다.

**Independent Test**: 참여자가 닉네임을 변경하거나 쪽지를 보냈을 때 관리자 모니터링 화면에 즉시 로그가 나타나는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 실시간 모니터링 대시보드에서, **When** 참여자가 닉네임을 변경하면, **Then** [기존 닉네임 -> 변경 닉네임] 형태의 로그가 즉시 노출된다.
2. **Given** 메시지 관리 화면에서, **When** 전체 쪽지 조회 버튼을 클릭하면, **Then** 파티 내에서 오간 모든 쪽지 내용이 시간순으로 표시된다.

---

### Edge Cases

- 파티 종료 시간이 시작 시간보다 앞선 경우 시스템은 에러를 발생시켜야 함.
- 실시간 공지 전송 시 네트워크 장애가 발생할 경우 재전송 또는 실패 알림을 표시해야 함.
- 예약 연동 시 중복된 전화번호나 데이터가 있을 경우 처리 정책(덮어쓰기 또는 제외)을 관리자가 선택할 수 있어야 함.
- 동일 앵커 URL로 여러 파티가 생성된 경우, 특정 시점에 '진행 중'인 파티는 오직 하나만 존재해야 하며 중복 활성화는 불가함.
- 참여자가 닉네임을 3회 초과하여 변경 시도 시 변경 버튼을 비활성화하고 안내 메시지를 노출해야 함.
- 관리자 권한 없는 사용자가 `/admin` 하위 경로 접근 시 즉시 세션 검증 후 로그인 페이지로 튕겨내야 함.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-000**: 시스템은 슈퍼 관리자를 위한 아이디/패스워드 기반의 로그인 기능을 제공해야 한다.
- **FR-001**: 시스템은 파티의 생성, 수정, 삭제 기능을 제공해야 한다.
- **FR-002**: 파티 목록은 페이징 및 검색 필터를 지원해야 하며, 경로는 `/admin/parties`를 사용한다.
- **FR-003**: 파티 상단에 단계별 진행 플로우(Flow Step) 가이드와 상세 매뉴얼 팝업을 제공해야 한다.
- **FR-004**: 관리자는 진행 중인 파티의 참여자에게 실시간으로 공지 메시지를 전송할 수 있어야 한다.
- **FR-005**: 관리자 상세 제어 페이지(`/admin/parties/[id]`)를 통해 파티의 **시작/종료 시간 및 정원(인원수)을 실시간으로 수정**할 수 있어야 한다. (FR-001 확장)
- **FR-006**: 관리자는 참여자 목록에서 **1차/2차 신청 상태를 개별적으로 관리(토글/승인)**하고, **큐피트/호감도 잔여 횟수를 조정(개별 및 전체 일괄 지급)**할 수 있어야 한다.
- **FR-014**: 파티 생성 및 수정 시, 호감도/큐피트의 기본 제공 횟수를 선택할 수 있는 **3종 프리셋(일반, 열정, 커스텀)** 기능을 제공해야 한다.
- **FR-007**: 시스템은 참여자의 모든 닉네임 변경 이력을 기록하고, 관리자가 **닉네임 변경 전후 이력**을 조회할 수 있는 뷰어를 제공해야 한다.
- **FR-008**: 관리자 대시보드 내에 모든 상호작용(쪽지, 호감도, SOS)이 실시간으로 흐르는 **통합 모니터링 피드**를 구축해야 한다.
- **FR-009**: 관리자는 닉네임 등록자 중 "2차 신청 미완료자"를 필터링하여 모아볼 수 있는 기능을 제공해야 한다.
- **FR-010**: 참여자 대시보드 상단에 **파티 종료까지 남은 시간(실시간 타이머)**을 초 단위로 표시해야 한다.
- **FR-011**: 참여자는 자신이 **수신하거나 발신한 모든 상호작용(쪽지, 호감도 등)의 상세 내역**을 조회할 수 있는 전용 보관함을 이용할 수 있어야 한다.
- **FR-012**: 랭킹 페이지에서 **본인의 현재 순위 및 정보를 시각적으로 강조(Highlight)**하여 표시해야 한다.
- **FR-013**: (제외) 랭킹 산정 가중치 수정 기능은 구현하지 않으며, 시스템 기본 설정값을 사용한다.

### Key Entities *(include if feature involves data)*

- **Party**: 파티 기본 정보(ID, 명칭, 시작/종료 시간, 최대 인원, 상태).
- **Participant**: 참여자 정보(닉네임, 횟수, 신청 상태, 닉네임 변경 이력).
- **Interaction**: 호감도/큐피트 발사 기록 (유형, 보낸이, 받는이, 시간).
- **Message**: 익명 쪽지 데이터 (내용, 보낸이, 받는이, 시간).
- **Announcement**: 관리자 전역 공지.
- **System Setting**: 랭킹 가중치, 시스템 전역 설정값.

### Constraints & Constitution

- **C-001**: 모든 문서, 코드 주석, 커밋 메시지는 **한글**로 작성하며, 작성자는 **ctxwing@gmail.com**으로 고정한다.
- **C-002**: 모든 시스템 시간 및 데이터는 **KST (UTC+9)** 설정을 따른다.
- **C-003**: 오직 **`main`** 브랜치만을 사용한다.
- **C-004**: UI 디자인 시 **Glassmorphism**과 **Dark Mode** 테마를 우선 적용하여 프리미엄 감성을 유지한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자 상세 제어 기능(시간/인원 수정)이 저장 후 1초 이내에 모든 참여자에게 반영됨.
- **SC-002**: 1,000명 이상의 동시 접속 환경에서도 실시간 모니터링 피드가 끊김 없이 업데이트됨.
- **SC-003**: 참여자가 자신의 모든 활동 내역을 3회 이내의 클릭으로 조회할 수 있는 UI 접근성 확보.
- **SC-004**: 1만 건 이상의 누적 데이터 상황에서도 관리자 필터링(미완료자 등) 결과가 0.5초 이내에 도출됨.
