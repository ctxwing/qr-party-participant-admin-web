# 프로젝트 요구사항 비교: A vs B

이 문서는 기존 프로젝트(A)와 새로운 게스트하우스 요구사항(B)을 비교 분석한 결과입니다.

## 1. 기본 정보 비교

| 항목 | 프로젝트 A (기존) | 프로젝트 B (게스트하우스) | 차이점 및 특징 |
| :--- | :--- | :--- | :--- |
| **프로젝트명** | QR 기반 파티 모바일웹 MVP | 게스트하우스 파티용 소통 웹 플랫폼 | B가 조금 더 범용적인 플랫폼 명칭 사용 |
| **예상 금액** | 2,500,000원 | 5,000,000원 | **B의 예산이 2배 높음** |
| **예상 기간** | 10일 | 30일 | **B의 기간이 3배 길음 (품질 중시)** |
| **클라이언트 위치** | 부산광역시 동구 | 서울특별시 강남구 | 지역적 차이 |
| **기획 상태** | 아이디어 단계 | 아이디어 단계 | 동일 |

## 2. 기능적 요구사항 비교

### 2.1 공통 기능 (거의 동일)
두 프로젝트 모두 QR 코드를 통한 입장과 실시간 상호작용을 핵심으로 하며, 아래 기능은 거의 100% 일치합니다.
- **실시간 정보**: 파티 남은 시간, 참여 인원수 표시
- **개인화**: 닉네임 설정/변경 (3회 제한), 본인 파티 신청 정보 확인
- **상호작용**: 쪽지 보내기, 큐피트 발사(2회), 호감도 주기(3회)
- **대시보드**: 내 현황판 (수신 개수 및 알림)
- **순위/요청**: 호감도 순위 표시, 노래/SOS 요청 기능

### 2.2 프로젝트 B만의 추가/차별화 포인트
프로젝트 B는 A의 기능을 기본으로 하되, 운영 효율성과 확장성에 더 비중을 두고 있습니다.
- **예약 시스템 연동**: 기존 홈페이지 예약 내역과의 연동 가능성 검토 (A에는 없음)
- **전체 공지 기능**: 채팅방 형태의 실시간 공지 발송 (A는 단순히 정보 표시 위주)
- **UI/UX 디자인 강조**: Figma, XD 등 디자인 원본 파일 제출을 명시 (A보다 디자인 퀄리티 기대치가 높음)
- **모니터링 강화**: 실시간 활동 모니터링 범위를 구체적으로 명시

## 3. 관리자 기능 비교

| 기능 구분 | 프로젝트 A (관리자) | 프로젝트 B (관리자) |
| :--- | :--- | :--- |
| **인원/시간 관리** | 파티 시간 설정, 인원수 수정 | 참석자 목록 조회 및 관리 |
| **신청 관리** | 1차/2차 신청 상태 및 필터링 | **기존 예약 내역 연동 (협의)** |
| **커뮤니케이션** | 쪽지 조회 (관리자 전용) | **전체 실시간 공지 발송 (채팅 형태)** |
| **로그 모니터링** | 닉네임 변경 로그 조회 | 사용자 활동 실시간 모니터링 |
| **설정 관리** | 큐피트/호감도 횟수 조정 | (명시되지 않았으나 유사할 것으로 예상) |

## 4. 종합 분석 및 제언

### 분석 결과
- **기능적 유사성**: 사용자 페이지의 핵심 기능은 프로젝트 A와 B가 거의 동일합니다. 이는 프로젝트 A의 코드 베이스를 프로젝트 B에 그대로 활용(Re-use)할 수 있음을 시사합니다.
- **차별점**: 프로젝트 B는 **'예약 연동'**과 **'실시간 공지'**라는 추가적인 백엔드 로직이 필요하며, 예산과 기간이 넉넉한 만큼 더 높은 수준의 **UI/UX 디자인**과 **시스템 안정성**을 요구합니다.

### 대응 전략
1. **코드 재사용**: A에서 개발된 QR 입장, 호감도, 큐피트, 랭킹 시스템을 모듈화하여 B에 이식합니다.
2. **추가 개발**: B의 핵심 차별점인 예약 시스템 연동 API 및 실시간 공지(WebSocket 기반 채팅 등) 기능을 추가 구현합니다.
3. **디자인 강화**: B의 높은 예산에 맞춰 인터페이스의 심미적 요소를 강화하고 반응형 최적화에 더 집중합니다.
