## 문제 원인 분석

### 원인 1: Better Auth 1.6.7의 DB 연결 API 변경
Better Auth 1.6.7은 Kysely 기반으로 재작성됨. `{ type: "postgres", url: "..." }` 불가.
`new PostgresDialect({ pool })` 필수.

### 원인 2: Kysely PostgresDialect는 `pg` 전용
- Kysely가 `pool.query(text, params)` 호출
- `postgres.js`는 tagged template literal만 지원 → `NOT_TAGGED_CALL`
- `pg-pool-adapter.ts`에서 `sql.unsafe()`로 래핑하여 우회

### 원인 3: Supabase Pooler SNI 문제 (핵심 미해결)
- `postgres.js`의 **모든 연결 시도**에서 `(ENOIDENTIFIER)` 발생
- 기존 `db.ts`의 Drizzle ORM 쿼리는 정상 작동
- `db.ts`의 client를 공유해도 `sql.unsafe()` 경로에서 동일 에러
- **Drizzle ORM은 tagged template literal을 사용**하여 postgres.js와 정상 통신
- **Kysely는 `.query(text, params)`를 사용**하므로 `sql.unsafe()` 필요
- `sql.unsafe()`가 새 연결을 생성할 때 SNI hostname이 누락되는 것으로 추정

### 원인 4: Turbopack + `pg` 호환 불가
- `pg` 패키지 설치해도 Turbopack에서 번들링 실패
- `serverExternalPackages`로도 해결 안 됨

### 기술 스택 호환성 매트릭스

| 조합 | 어댑터 초기화 | DB 연결 | Turbopack | 총평 |
|------|-------------|---------|-----------|------|
| `{ type, url }` 구방식 | ❌ | - | - | 미지원 |
| Kysely + postgres.js 직접 | ✅ | ❌ NOT_TAGGED_CALL | - | API 불일치 |
| Kysely + pg.Pool | ✅ | ❌ SELF_SIGNED_CERT | ❌ 번들링 실패 | 3중 불가 |
| Kysely + pg-pool-adapter (새 인스턴스) | ✅ | ❌ ENOIDENTIFIER | ✅ | SNI 문제 |
| Kysely + pg-pool-adapter (db.ts 공유) | ✅ | ❌ ENOIDENTIFIER | ✅ | SNI 문제 |

### 미해결의 근본 원인
**Supabase Pooler의 SNI/Tenant 식별 요구사항과 `postgres.js`의 처리 방식 간의 충돌**이 근본 원인이었습니다.

1. **Supabase Pooler(Transaction Mode, 6543)**는 연결 시 해당 연결이 어느 프로젝트(Tenant)의 것인지 식별해야 합니다.
2. 기존에 사용된 연결 문자열: `postgresql://postgres:패스워드@aws-0-ap-northeast-2.pooler.supabase.com:6543/postgres?sslmode=require`
3. 위 연결 문자열을 사용하면, Pooler는 TLS **SNI (Server Name Indication)** 정보에 의존하여 Tenant를 식별하려고 시도합니다.
4. 그러나 `postgres.js` 라이브러리는 내부 연결(`connection.js`) 생성 시, 호스트명이 IP 주소가 아닌 경우 SNI `servername`을 자동으로 설정하지만, Transaction Pooler 환경과 맞물려 `sql.unsafe()` 등 특정 쿼리 방식 사용 시 이 SNI 정보가 Supabase가 요구하는 형태로 완벽하게 전달/식별되지 않는 현상이 발생했습니다. (결과: `ENOIDENTIFIER`)

## 해결 방법

**연결 문자열(DATABASE_URL) 형식을 변경하여 SNI에 의존하지 않고 명시적으로 Tenant를 식별하도록 수정**했습니다.

Supabase Pooler는 `postgres.[project_ref]` 형태의 username을 지원하며, 이 방식을 사용하면 SNI 없이도 Tenant를 식별할 수 있습니다. 또한, 기존 문서(`IPv6_대처방안.md`)를 확인한 결과 실제 Pooler 호스트가 `aws-1` 노드임을 확인했습니다.

### 최종 조치 내역
`.env.local` 파일의 `DATABASE_URL`을 올바른 Pooler 형식으로 변경했습니다.

**변경 전 (SNI 의존 방식, 에러 발생)**
```env
DATABASE_URL=postgresql://postgres:%40Supabase01%40@aws-0-ap-northeast-2.pooler.supabase.com:6543/postgres?sslmode=require
```

**변경 후 (명시적 Tenant 식별 방식, 정상 동작)**
```env
DATABASE_URL=postgresql://postgres.hlbgedbgycamzvbbykdc:%40Supabase01%40@aws-1-ap-northeast-2.pooler.supabase.com:6543/postgres?sslmode=require
```

### 결과
- `db.ts`의 Drizzle ORM 쿼리 정상 작동
- `postgres.js`의 기본 Tagged Template Literal 쿼리 정상 작동
- Better Auth(Kysely 어댑터)가 사용하는 `pg-pool-adapter`의 `sql.unsafe()` 쿼리도 정상 작동
- 관리자 로그인 기능(Better Auth 세션 검증) 500 에러 해결
