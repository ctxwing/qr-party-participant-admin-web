## 문제 원인 분석

### 원인 1: Better Auth 1.6.7의 DB 연결 API 변경
Better Auth 1.6.7은 Kysely 기반으로 재작성됨. `{ type: "postgres", url: "..." }` 불가.
`new PostgresDialect({ pool })` 필수.

### 원인 2: Kysely PostgresDialect는 `pg` 전용
- Kysely가 `pool.query(text, params)` 호출
- `postgres.js`는 tagged template literal만 지원 → `NOT_TAGGED_CALL`
- `pg-pool-adapter.ts`에서 `sql.unsafe()`로 래핑하여 우회

### 원인 3: Supabase Pooler SNI 문제 (핵심 미해결)
- `postgres.js`의 **모든 연결 시도**에서 `(ENOIDENTIFIER)` 발생
- 기존 `db.ts`의 Drizzle ORM 쿼리는 정상 작동
- `db.ts`의 client를 공유해도 `sql.unsafe()` 경로에서 동일 에러
- **Drizzle ORM은 tagged template literal을 사용**하여 postgres.js와 정상 통신
- **Kysely는 `.query(text, params)`를 사용**하므로 `sql.unsafe()` 필요
- `sql.unsafe()`가 새 연결을 생성할 때 SNI hostname이 누락되는 것으로 추정

### 원인 4: Turbopack + `pg` 호환 불가
- `pg` 패키지 설치해도 Turbopack에서 번들링 실패
- `serverExternalPackages`로도 해결 안 됨

### 기술 스택 호환성 매트릭스

| 조합 | 어댑터 초기화 | DB 연결 | Turbopack | 총평 |
|------|-------------|---------|-----------|------|
| `{ type, url }` 구방식 | ❌ | - | - | 미지원 |
| Kysely + postgres.js 직접 | ✅ | ❌ NOT_TAGGED_CALL | - | API 불일치 |
| Kysely + pg.Pool | ✅ | ❌ SELF_SIGNED_CERT | ❌ 번들링 실패 | 3중 불가 |
| Kysely + pg-pool-adapter (새 인스턴스) | ✅ | ❌ ENOIDENTIFIER | ✅ | SNI 문제 |
| Kysely + pg-pool-adapter (db.ts 공유) | ✅ | ❌ ENOIDENTIFIER | ✅ | SNI 문제 |

### 미해결의 근본 원인
**Better Auth(Kysely) + Supabase Transaction Mode Pooler + Turbopack** 의 3중 호환성 문제:
1. Kysely는 `.query()` API 필요 → `pg` 또는 `sql.unsafe()` 래핑 필요
2. `pg`는 Turbopack에서 번들링 불가
3. `sql.unsafe()`는 Supabase Pooler에서 SNI 문제 발생
4. Drizzle ORM의 tagged template literal 방식만 Supabase Pooler에서 정상 작동
