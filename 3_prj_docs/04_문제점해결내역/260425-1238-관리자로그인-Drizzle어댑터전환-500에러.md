# 관리자 로그인 무반응 (500 Internal Server Error)

## 발생 시각
- 2026-04-25 12:38 (커밋 `3b94808` 푸시 직후)

## 문제 현상
- 관리자 로그인 페이지에서 [DEV: ADMIN AUTO-LOGIN] 버튼 클릭 시 아무 응답 없음
- `POST /api/auth/sign-in/email` → 500 Internal Server Error
- 서버 로그: `PostgresError: (ENOIDENTIFIER) no tenant identifier provided (external_id or sni_hostname required)`
- 이후 원복 후에도 `BetterAuthError: Failed to initialize database adapter` 에러 지속

## 문제 원인 분석

### 직접 원인
커밋 `3b94808`에서 `auth.ts`의 DB 연결 방식을 Better Auth 내장 PostgreSQL 드라이버에서 **Drizzle 어댑터**로 전환함.

| 항목 | 변경 전 (정상) | 변경 후 (에러) |
|------|---------------|---------------|
| auth.ts DB 설정 | `database: { type: "postgres", url: ... }` | `database: drizzleAdapter(db, { provider: "pg", schema: ... })` |
| DB 드라이버 | Better Auth 내장 PostgreSQL 드라이버 | `postgres.js` (db.ts의 클라이언트) |
| Supabase Pooler | 정상 처리 | SNI hostname 누락으로 tenant 식별 실패 |

### 근본 원인
- Better Auth의 **내장 PostgreSQL 드라이버**는 Supabase Transaction Mode Pooler (포트 6543) 연결 시 SSL SNI를 자동으로 올바르게 설정
- 반면 `postgres.js` 라이브러리를 통한 Drizzle 어댑터는 `{ prepare: false }` 설정만으로는 SNI hostname을 Supabase Pooler가 요구하는 형태로 전달하지 못함
- Supabase Pooler는 SSL 연결 시 SNI 또는 `external_id`로 테넌트를 식별하는데, 이 정보가 누락되어 `(ENOIDENTIFIER)` 에러 발생

### 2차 에러: Failed to initialize database adapter
- `auth.ts`를 원래 코드로 복원해도 동일 에러 지속
- **원인**: 커밋 `3b94808`에서 `bun add @better-auth/drizzle-adapter` 실행 시 `package.json`과 `bun.lock`이 변경됨
- `@better-auth/drizzle-adapter@1.6.9`가 추가되면서, Better Auth 내부 의존성 해석이 변경됨
- 코드 원복만으로는 부족하고, **의존성도 마지막 정상 커밋(`0015489`) 기준으로 복원 필요**

### 잘못된 대응 (시간 낭비)
1. 문제를 "테이블 미생성"으로 오진 → DDL 실행 시도 (Pooler에서 동일 SNI 에러)
2. Supabase PAT로 Management API 시도 → 토큰 만료 (`Unauthorized`)
3. Supabase SQL Editor 수동 실행 제안 → 근본 원인이 아님
4. `pg` 패키지 추가 설치 시도 → 이 프로젝트는 bun + `postgres.js` 기반, 불필요한 패키지 추가
5. `auth.ts` 코드만 원복 → `package.json`/`bun.lock` 변경을 간과하여 2차 에러 발생

## 해결 방법

### 최종 조치: 코드 + 의존성 완전 원복
1. `auth.ts`, `db.ts`, `schema.ts` → 마지막 정상 커밋(`0015489`) 기준으로 복원
2. `package.json`, `bun.lock` → `git checkout 0015489 -- package.json bun.lock` 으로 복원
3. `bun install` 재실행 → 의존성 원래 상태로 복구
4. 불필요 파일 제거: `drizzle.config.ts`, `setup-db` 임시 API route

## 교훈 및 향후 주의사항

1. **코드 원복 시 의존성도 함께 원복**: `package.json`, `bun.lock`, `node_modules`까지 모두 원래 상태로 돌려야 함. 코드만 원복하면 의존성 변경으로 인한 2차 에러 발생 가능.

2. **되던 게 안 되면 가장 최근 변경을 먼저 의심**: "몇 시간 전까지 되던 것"이 안 되면 원인 분석보다 **롤백을 먼저** 실행.

3. **에러 메시지 정확히 읽기**: `(ENOIDENTIFIER)`는 테이블 미존재가 아니라 **연결 자체의 인증 문제**. `Failed to initialize database adapter`는 **의존성/드라이버 문제**. 에러 메시지로부터 원인을 좁혀야 함.

4. **Supabase Pooler + 외부 ORM 어댑터 조합 주의**: Better Auth를 Supabase Pooler와 함께 사용할 때는 내장 드라이버(`type: "postgres"`)를 사용. Drizzle 어댑터는 SNI/tenant 식별 문제 발생 가능.

5. **프로젝트 기술 스택 준수**: bun 프로젝트에 `pg`(node-postgres)를 임의로 추가하지 않음. 기존 `postgres.js` 기반을 유지.

6. **Better Auth + Supabase 권장 설정**:
   ```typescript
   // 올바른 설정 (내장 드라이버)
   betterAuth({
     database: {
       type: "postgres",
       url: process.env.DATABASE_URL!,
     },
   })
   
   // 피해야 할 설정 (Drizzle 어댑터 + Supabase Pooler)
   betterAuth({
     database: drizzleAdapter(db, { provider: "pg", schema: ... }),
   })
   ```
