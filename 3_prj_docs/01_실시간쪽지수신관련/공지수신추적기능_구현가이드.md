# 공지 수신 추적 기능 구현 가이드

## 1. 프로젝트 특성 반영

- **파티 기간**: 1-2일 (짧은 기간)
- **멤버 구성**: 파티마다 새로운 참여자 (순환 구조)
- **데이터 특성**: 활성 파티 데이터만 중요, 과거 파티는 정리 가능
- **결론**: 1년 후 데이터 폭증은 **이슈가 아님** ✓

---

## 2. 필요한 DB 구조 (최소)

### announcement_reads 테이블
```sql
CREATE TABLE announcement_reads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  announcement_id UUID NOT NULL REFERENCES announcements(id) ON DELETE CASCADE,
  participant_id UUID NOT NULL REFERENCES participants(id) ON DELETE CASCADE,
  read_at TIMESTAMP NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(announcement_id, participant_id)
);

-- 인덱싱
CREATE INDEX idx_announcement_reads_announcement ON announcement_reads(announcement_id);
CREATE INDEX idx_announcement_reads_read_at ON announcement_reads(read_at);
```

**설명:**
- `read_at = NULL`: 아직 읽지 않음
- `read_at = timestamp`: 읽음
- 파티 종료 후 해당 파티의 공지 데이터 삭제 가능

## 3. API 변경사항

### 공지 발송 시 (POST /api/admin/announcements)
```typescript
// 1. 공지 저장
const announcement = await supabase
  .from('announcements')
  .insert({ content, type })
  .select()
  .single();

// 2. 활성 파티 참여자 조회
const participants = await supabase
  .from('participants')
  .select('id')
  .eq('party_id', currentPartyId);  // 현재 활성 파티만

// 3. 읽음 기록 배치 생성
const reads = participants.map(p => ({
  announcement_id: announcement.id,
  participant_id: p.id
}));

await supabase.from('announcement_reads').insert(reads);
```

**성능:**
- 100명: < 1초
- 1000명: < 2초

## 6. RLS 정책 (필수)

```sql
-- 참여자는 자신의 읽음 기록만 조회/수정 가능
ALTER TABLE announcement_reads ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own reads"
  ON announcement_reads
  FOR SELECT
  USING (participant_id = auth.uid());

CREATE POLICY "Users can update own reads"
  ON announcement_reads
  FOR UPDATE
  USING (participant_id = auth.uid());

-- 관리자는 모두 조회 가능
CREATE POLICY "Admins can view all reads"
  ON announcement_reads
  FOR SELECT
  USING (auth.jwt() ->> 'role' = 'admin');
```

## 7. 파티 종료 시 데이터 정리

```sql
-- 파티 종료 시 해당 파티 관련 공지와 읽음 기록 삭제
DELETE FROM announcement_reads 
  WHERE announcement_id IN (
    SELECT id FROM announcements 
    WHERE party_id = $1
  );

DELETE FROM announcements WHERE party_id = $1;
```

## 10. 참고

- 파티 기간이 짧아서 **일반적인 대량 데이터 문제는 발생하지 않음**
- 필요시 월 1회 정기적으로 3개월 이상 파티 데이터 아카이빙 추천
- 현재 announcements 테이블에 `party_id` 추가 필요 (공지가 어느 파티인지 구분)
