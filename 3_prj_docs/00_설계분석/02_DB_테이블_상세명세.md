# 📊 DB 테이블 상세 명세 (Database Specification)

본 문서는 **QR 파티 관리 시스템**의 데이터베이스 구조와 각 테이블의 역할을 정의합니다.

## 1. 참여자 관리 (Participants)
참여자의 고유 정보를 관리하며, 닉네임 변경 시에도 고유 ID를 통해 이력을 추적합니다.

| 컬럼명 | 타입 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **id** | UUID | 참여자 고유 식별자 | Primary Key (고정값) |
| **nickname** | TEXT | 참여자 닉네임 | 중복 불가 (변경 가능) |
| **created_at** | TIMESTAMPTZ | 가입 일시 | 자동 생성 |
| **last_participated_at** | TIMESTAMPTZ | 최근 활동 일시 | 최근 접속/기능 사용 시 갱신 |
| **cupid_count** | INTEGER | 사용 가능한 큐피트 횟수 | 기본값: 3 |
| **like_count** | INTEGER | 보낼 수 있는 호감 표시 횟수 | 기본값: 10 |
| **is_first_applied** | BOOLEAN | 1차 신청 여부 | 기본값: FALSE |
| **is_second_applied** | BOOLEAN | 2차 신청 여부 | 기본값: FALSE |

---

## 2. 행사 세션 (Party Sessions)
파티 행사의 시작과 종료, 그리고 행사별 명칭을 관리합니다.

| 컬럼명 | 타입 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **id** | BIGINT | 세션 고유 번호 | Primary Key (자동 증가) |
| **title** | TEXT | 행사명 (예: 게스트하우스 1차 파티) | 시작 시 입력 |
| **status** | TEXT | 세션 상태 (READY, ONGOING, FINISHED) | 현재 상태 관리 |
| **start_time** | TIMESTAMPTZ | 행사 시작 시간 | - |
| **end_time** | TIMESTAMPTZ | 행사 종료 시간 | 종료 시 자동 기록 |
| **created_at** | TIMESTAMPTZ | 데이터 생성 일시 | - |

---

## 3. 메시지 이력 (Messages)
참여자 간에 송수신된 쪽지 데이터를 보관합니다.

| 컬럼명 | 타입 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **id** | UUID | 메시지 고유 식별자 | Primary Key |
| **sender_id** | UUID | 발신자 고유 ID | Foreign Key (participants.id) |
| **receiver_id** | UUID | 수신자 고유 ID | Foreign Key (participants.id) |
| **content** | TEXT | 메시지 내용 | - |
| **created_at** | TIMESTAMPTZ | 발송 일시 | - |

---

## 4. 실시간 요청 (Alerts)
SOS 요청 및 노래 신청 등 관리자에게 전달되는 실시간 요청을 관리합니다.

| 컬럼명 | 타입 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **id** | BIGINT | 알람 고유 번호 | Primary Key |
| **participant_id** | UUID | 요청자 고유 ID | Foreign Key (participants.id) |
| **type** | TEXT | 요청 타입 (SOS, MUSIC) | - |
| **message** | TEXT | 요청 내용/노래 제목 | - |
| **is_resolved** | BOOLEAN | 처리 여부 | 기본값: FALSE |
| **created_at** | TIMESTAMPTZ | 요청 일시 | - |

---

## 💡 설계 핵심 포인트
1.  **UUID 추적**: `nickname`은 사용자가 언제든 바꿀 수 있지만, 모든 데이터는 고유 `id(UUID)`를 기반으로 연동되어 있어 이력 추적이 완벽하게 보장됩니다.
2.  **행사 이력화**: `party_sessions` 테이블에 `title`과 `end_time`을 추가함으로써, 단발성 행사가 아닌 과거 행사의 기록을 통계화할 수 있는 기반을 마련했습니다.
3.  **관리자 가시성**: `last_participated_at` 필드를 통해 유령 회원과 실제 활성 회원을 구분하여 관리할 수 있습니다.
